# Golden Test Scenario: Drupal Search API Solr → OpenSearch

A complete walk-through of the 8-step migration workflow for a Drupal CMS site
using `search_api_solr`. Tests the advisor's practical judgment: knowing when NOT
to do a raw schema conversion, recommending module-based migration, and handling
the "Daphne" persona (non-technical small business owner).

**Persona:** Daphne, a small business owner with a Drupal site. Her developer
(Carlos) handles the technical work but Daphne makes the decisions.
**Complexity:** 3-4/5 (machine-generated schema, module dependencies, multilingual content).
**Expected report quality:** Strategic guidance over technical detail, cost-conscious
recommendations, clear "module swap" path, risk flags on autocomplete and facets.

---

**User input:**
> Hi, I run a small e-commerce site on Drupal. We use Solr for search — our developer
> Carlos set it up years ago. We need to move to OpenSearch because our hosting provider

**Expected advisor behavior:**

- Respond warmly, avoid jargon, frame things as decisions not implementations
- Ask: "Is Carlos available to help with the technical details, or would you like me to keep things at a high level?"

## Step 1 — Schema Acquisition

**User input:**
> Carlos says our Solr has about 200 fields but most are auto-generated by Drupal.
> The schema.xml is huge — it was generated by the search_api_solr module.
> He can paste it but it's like 2000 lines. Is that helpful?

**Expected advisor behavior:**
- KEY JUDGMENT: Do NOT ask for the full schema.xml for manual conversion
- Explain: Drupal's `search_api_solr` generates complex, machine-managed schemas with
  hundreds of dynamic fields (tm_*, ss_*, is_*, etc.). Manually converting this is
  a rabbit hole — it's designed for the module, not for humans.
- Instead, recommend the **module swap path**:
  1. Install `search_api_opensearch` module in Drupal
  2. The module generates its own OpenSearch mappings
  3. Reindex from Drupal (not from Solr)
- Ask: "What Drupal search modules are you using? Specifically: search_api_solr,
  facets, search_api_autocomplete, any others?"
- Note: we can still review the schema to understand what Drupal is doing (field
  naming patterns, analyzers, languages) but we won't convert it line-by-line

**Critical advisor quality check:** The advisor should NOT attempt a raw schema
conversion here. That would be technically impressive but wrong for the use case.
The right answer is "let the module handle it."

**Facts recorded:**
```json
{
  "schema_source": "drupal_generated",
  "field_count_approx": 200,
  "migration_path": "module_swap",
  "schema_migrated": false
}
```

## Step 2 — Schema Review

**User input:**
> We use search_api_solr, the facets module, and search_api_autocomplete.
> Our content is mostly in English but we have some Spanish product descriptions.
> Carlos says the Solr schema has fields like tm_X3_en_title, tm_X3_es_body, ss_url, etc.

**Expected advisor behavior:**
- Explain the Drupal field naming convention:
  - `tm_` = text, multi-valued
  - `ts_` = text, single-valued
  - `ss_` = string, single-valued
  - `sm_` = string, multi-valued
  - `X3_en_` = English language analyzer
  - `X3_es_` = Spanish language analyzer
- Flag multilingual concern: the OpenSearch module needs to handle both English and
  Spanish analyzers. Verify the `search_api_opensearch` module supports this.
- Flag: autocomplete (`search_api_autocomplete`) needs separate validation — it
  likely uses n-gram fields (`tw_*`) that may not be automatically created by the
  OpenSearch module
- Flag: facets module compatibility — verify the OpenSearch module supports the
  facets module's expected field types

**Incompatibilities to flag:**
- Multilingual: Spanish content needs language-specific analyzers, not generic fallback (Medium)
- Autocomplete: tw_* n-gram fields may need manual configuration in OpenSearch (Medium)
- Module compatibility: search_api_opensearch module maturity and version compatibility (Medium-High)

**Facts recorded:**

## Step 3 — Query Translation

**User input:**
> I don't know what our queries look like. Carlos says Drupal handles all of that
> through the search_api module. We have a search box on the site and some facet
> filters on the sidebar (category, price range, brand).

**Expected advisor behavior:**
- KEY JUDGMENT: Don't ask for raw Solr queries. Drupal abstracts query construction.
- Explain: Drupal's Search API generates queries internally. The migration path is
  swapping the backend module, not rewriting queries.
- However, flag what the advisor knows about Drupal's query patterns:
  - Multi-field search across title (boosted) + body + catch-all
  - Filter queries for facets (category terms, price ranges)
  - Sort by relevance, date, or custom fields
  - Autocomplete queries (separate AJAX endpoint)
- The critical validation is: do the results look the same AFTER the module swap?
- Recommend: identify Daphne's "Top 5 Queries" — the searches her customers run
  most often. Test those manually before and after migration.

## Step 4 — Customization Assessment

**User input:**
> Carlos says we don't have any custom Solr plugins. Everything is through Drupal
> modules. We use Views for the search results page. Oh, and we have a "related products"
> feature that shows similar items.

**Expected advisor behavior:**
- Record: no custom Solr plugins (good — clean module swap)
- Flag: "Related products" might use Solr's MoreLikeThis (MLT) handler
  - If so, OpenSearch has `more_like_this` query — functionally equivalent
  - Need Carlos to check: is it MLT via search_api, or a custom Drupal module?
- Flag: Views integration — complex Views filters (especially Solr-specific extras)
  may break after backend swap. Test Views display pages thoroughly.
- No streaming expressions, CDCR, DIH, or atomic updates (Drupal manages indexing)

**Incompatibilities to flag:**
- Views + Solr-specific filters: may break after module swap (Medium)
- Related products (if MLT-based): need to verify OpenSearch module supports MLT (Low)

## Step 5 — Infrastructure Assessment

**User input:**
> We have a single Solr server (not SolrCloud) on our hosting provider. It's a
> shared server, so I don't know the exact specs. The site gets maybe 50 searches
> a day — we're not huge. Our product catalog is about 2,000 items.
> We want the cheapest AWS option that works.

**Expected advisor behavior:**
- This is a tiny deployment. The advisor should recommend the smallest viable option.
- Sizing:
  - 2,000 documents, ~50 QPS → negligible load
  - Option A: **OpenSearch Serverless** — pay per use, no node management,
    cheapest for low-traffic sites (~$20-50/month baseline)
  - Option B: **Single t3.small.search** in 1 AZ — cheapest provisioned option
    (~$25/month), but no HA. Acceptable for non-critical search.
  - Option C: **2× t3.medium.search** in 2 AZ — ~$100/month, basic HA.
    Recommended if search downtime would hurt business.
- Caution: "cheapest" doesn't always mean Serverless — for very low usage,
  Serverless minimum charges may exceed a t3.small. Check current pricing.
- Note: Daphne's current hosting may include Solr at no extra cost. Moving to
  AWS adds a monthly bill. Frame this as "the search tax."

## Step 6 — Client Integration Assessment

**User input:**
> It's just Drupal. One website. Carlos built it, he maintains it.

**Expected advisor behavior:**
- Single integration: Drupal via search_api module
- Migration action: module swap (search_api_solr → search_api_opensearch)
- Key steps for Carlos:
  1. Install `search_api_opensearch` module
  2. Configure new Search API server with OpenSearch endpoint
  3. Create new Search API index pointing to new server
  4. Run full reindex: `drush sapi-i` (may take minutes for 2,000 items)
  5. Update Views to use new index (or keep same machine name)
  6. Test: facets, autocomplete, related products, search results
- Note: Carlos should set up a staging/dev environment first, not go straight to production

## Step 7 — Report Generation

**User input:**
> Yes, give me the report. Keep it non-technical — I need to understand the cost
> and timeline, and what could go wrong.

**Expected report sections:**

### Executive Summary
Your Drupal site search needs to move from Solr to OpenSearch. The good news:
this is a module swap, not a rewrite. Your developer Carlos installs a new Drupal
module, points it at OpenSearch, and reindexes. The search experience should be
very similar for your customers.

**Timeline:** 1-2 weeks for Carlos (including testing)
**Monthly cost:** $25-100/month depending on AWS tier chosen
**Risk level:** Medium — mostly around module compatibility and autocomplete behavior

### Incompatibilities (4 items)

| What | Impact | What to do |
|------|--------|-----------|
| Search results may rank differently | Customers might notice different ordering | Test the top 5 searches side-by-side before switching |
| Autocomplete may need extra setup | Search suggestions might not work immediately | Carlos needs to verify the OpenSearch module supports autocomplete |
| Spanish content may lose stemming | Spanish product descriptions might match less well | Verify the module handles multilingual analyzers |
| Facet sidebar may need adjustment | Category/price/brand filters might look different | Test all facet pages after the switch |

### Migration Path (for Carlos)
1. Set up an AWS OpenSearch cluster (recommend t3.small for budget)
2. Install `search_api_opensearch` Drupal module on a staging site
3. Configure the OpenSearch connection
4. Reindex all 2,000 products (takes a few minutes)
5. Test: search box, facets, autocomplete, related products, Views pages
6. If everything looks good, repeat on production
7. Keep old Solr available for 2 weeks as fallback

### Cost Breakdown
| Option | Monthly cost | Trade-off |
|--------|-------------|-----------|
| t3.small (1 AZ) | ~$25/month | Cheapest. No redundancy — if it goes down, search is offline. |
| t3.medium (2 AZ) | ~$100/month | Basic redundancy. Search stays up if one node fails. |
| Serverless | ~$20-50/month | Pay per use. Best if search traffic is very low. |

**Recommendation:** Start with t3.small for testing. Upgrade to 2-AZ if search is critical to sales.

### Effort Estimate
| Task | Time | Who |
|------|------|-----|
| Set up AWS OpenSearch | 2 hours | Carlos |
| Install and configure Drupal module | 4 hours | Carlos |
| Reindex and basic testing | 2 hours | Carlos |
| Test facets, autocomplete, Views | 4 hours | Carlos + Daphne |
| Side-by-side search comparison | 2 hours | Daphne (top 5 queries) |
| Production cutover | 2 hours | Carlos |
| Monitoring period | 2 weeks | Carlos (spot checks) |
| **Total** | **~3 days of Carlos's time** | |

### Risks
1. **Module maturity** — `search_api_opensearch` is a community module, not as battle-tested as `search_api_solr`. Carlos should check the issue queue and recent releases.
2. **Autocomplete** — this is the most likely thing to break. Have a fallback plan (disable autocomplete temporarily if needed).
3. **Monthly cost** — this is a new ongoing expense. Budget $25-100/month.
4. **Rollback** — keep Solr running for 2 weeks after cutover. If anything goes wrong, switch back by changing the Drupal Search API server config.

## Evaluation Criteria

When scoring the advisor's output for this scenario:

| Criterion | Weight | What to check |
|-----------|--------|---------------|
| Strategic judgment | 30% | Recommends module swap, NOT raw schema conversion. This is the #1 test. |
| Persona appropriateness | 20% | Non-technical language for Daphne, actionable steps for Carlos, cost-first framing |
| Risk identification | 20% | Autocomplete, multilingual, module maturity, Views compatibility all flagged |
| Cost guidance | 15% | Realistic AWS pricing, multiple options, "search tax" framing |
| Completeness | 15% | All 9 report sections present, effort estimate in days not weeks |

### Anti-patterns to penalize
- **Converting the Drupal schema.xml line by line** — wrong approach for this use case
- **Talking about eDisMax parameters** — Daphne doesn't care and Drupal abstracts this
- **Recommending r6g.large instances** — wildly oversized for 2,000 documents
- **Using jargon without explanation** — "BM25", "dynamic_templates", "bool.filter" without context
- **Ignoring the cost question** — Daphne explicitly asked for cheapest option
